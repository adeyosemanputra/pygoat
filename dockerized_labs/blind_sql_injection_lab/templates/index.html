What is Blind SQL Injection?

Blind SQL injection happens when the application does not reveal raw database errors, but still behaves differently based on whether an injected SQL condition is true or false. Attackers can infer data one character at a time by observing response messages.

Blind SQL usually appears when:

  • Input is directly concatenated into SQL queries
  • Detailed SQL errors are hidden from users
  • Error handling and logging are inconsistent
  • Application responses leak small boolean clues

Lab Challenge

Your goal is to log in as the admin user. The password is not shown, database errors are not exposed, and you must infer correct values through server responses.

How to approach this lab:

  • Open the lab login page and use username admin
  • Intercept requests with Burp Suite (or similar proxy)
  • Send boolean-based payloads and observe response changes
  • When a condition is true, the app hints: You are going right
  • Extract the password one character at a time (alphanumeric)

Expected Behavior

Unlike classic SQL injection labs, this one does not reveal SQL tracebacks. You only get behavior-based clues, which simulates real-world blind SQL testing.

Mitigation

  • Use parameterized queries (prepared statements)
  • Implement strict input validation
  • Use centralized error handling and safe logging
  • Do not leak behavioral signals in authentication responses
  • Add rate limiting and brute-force detection

Insecure Direct Object Reference (IDOR)

What is IDOR?

Insecure Direct Object Reference (IDOR) is a type of access control vulnerability that occurs when an application uses user-supplied input to directly access objects (like database records, files, or user profiles).

The vulnerability exists when the application fails to verify that the user is authorized to access the specific object they're requesting.

Common IDOR Patterns

  • /api/user/123 - Sequential user IDs
  • /download?file=report_456.pdf - Document IDs
  • /order/789/details - Order numbers
  • /account/balance?id=101 - Account references

Why is IDOR Dangerous?

  • Data Breach: Access to other users' personal information
  • Privacy Violation: Exposure of sensitive data (SSN, salary, medical records)
  • Horizontal Privilege Escalation: Acting as another user
  • Mass Harvesting: Attackers can iterate through IDs to collect all data

Lab Challenge

This lab simulates a company's employee directory system. Each employee can view their own profile through the API endpoint /api/user/{user_id}.

Your Mission: Access the admin user's sensitive data (SSN and salary) without having admin credentials.

Available Credentials

Username Password User ID Role
john john123 1 User
jane jane123 2 User
admin ??? 3 Admin

Hints

  1. Login as john using the credentials above
  2. Notice your User ID is shown after login
  3. The API endpoint is /api/user/{user_id}
  4. What happens if you change the user_id to a different number?
  5. Can you access /api/user/3 while logged in as john?

Mitigation Strategies

  • Authorization Checks: Always verify the user has permission to access the requested resource
  • Indirect References: Use random tokens instead of sequential IDs
  • Access Control Lists: Implement proper ACLs for resource access
  • Session Validation: Compare session user with requested resource owner
  • Logging: Log access attempts for security monitoring

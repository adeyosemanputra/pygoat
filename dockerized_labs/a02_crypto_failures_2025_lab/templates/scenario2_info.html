Scenario 2: Insecure Cryptographic Storage

Vulnerability: Hardcoded encryption keys in source code (CWE-259, CWE-321)

The Problem

Storing encryption keys directly in application code is one of the most critical cryptographic failures. If an attacker gains access to the source code (through repository leaks, insider threats, or other means), all encrypted data becomes instantly compromised.

Common Mistakes

  • Hardcoding keys directly in source files
  • Committing keys to version control systems (Git, SVN)
  • Storing keys in configuration files within the codebase
  • Using the same key across all environments (dev, staging, prod)

Real-World Example: Uber Data Breach (2016)

Uber suffered a massive breach affecting 57 million users. Attackers found AWS credentials hardcoded in a private GitHub repository. This allowed them to access S3 buckets containing sensitive user data.

Vulnerable vs Secure Code

❌ Vulnerable – Hardcoded Encryption Key

# VULNERABLE: Hardcoded encryption key
ENCRYPTION_KEY = 'ZmDfcTF7_60GrrY167zsiPd67pEvs0aGOv2oasOM1Pg='

from cryptography.fernet import Fernet
cipher = Fernet(ENCRYPTION_KEY)
encrypted_data = cipher.encrypt(b"sensitive_info")

# If attacker gets source code, ALL encrypted data is compromised!

✅ Secure – Environment Variables / Key Vaults

# SECURE: Use environment variables
import os
from cryptography.fernet import Fernet

# Key stored in environment variable or key vault
ENCRYPTION_KEY = os.environ['ENCRYPTION_KEY']
cipher = Fernet(ENCRYPTION_KEY)

# Alternative: Use cloud key management services
# - AWS KMS (Key Management Service)
# - Azure Key Vault
# - Google Cloud KMS
# - HashiCorp Vault

Best Practices

  • Store keys in environment variables or dedicated key vaults
  • Never commit keys to version control (.gitignore sensitive files)
  • Rotate keys regularly
  • Use different keys for different environments
  • Implement proper key access controls

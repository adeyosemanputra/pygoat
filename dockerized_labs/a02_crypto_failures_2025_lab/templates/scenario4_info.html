Scenario 4: CBC Bit-Flipping Attack

Start Lab → ← Back to Home

The Vulnerability

CBC (Cipher Block Chaining) is a block cipher mode that was widely used for encryption. While CBC provides confidentiality, it does NOT provide integrity or authentication by itself.

In CBC mode, each block of plaintext is XORed with the previous ciphertext block before encryption. This creates a dependency chain, but it also creates an exploitable property:

Bit-Flipping Attack: An attacker can modify bits in the ciphertext to cause predictable changes in the decrypted plaintext without knowing the encryption key!

How CBC Mode Works

Encryption:
C[0] = Encrypt(P[0] XOR IV)
C[1] = Encrypt(P[1] XOR C[0])
C[2] = Encrypt(P[2] XOR C[1])
...

Decryption:
P[0] = Decrypt(C[0]) XOR IV
P[1] = Decrypt(C[1]) XOR C[0]
P[2] = Decrypt(C[2]) XOR C[1]
...

Key Observation: When decrypting block N, the previous ciphertext block N-1 is XORed with the decrypted result. If we modify C[N-1], it directly affects P[N] through XOR!

The XOR Property

XOR has a useful property for attackers:

A XOR B XOR B = A  (XOR is reversible)

If: P = Decrypt(C) XOR IV
Then: P XOR X = Decrypt(C) XOR (IV XOR X)

By flipping bits in IV (or previous ciphertext block),
we can flip corresponding bits in the plaintext!
Attack Vector:

If we know (or can guess) part of the plaintext, we can:

  1. Calculate what bits to flip in the previous ciphertext block
  2. Flip those bits to change the plaintext to our desired value
  3. The application decrypts it and trusts the modified data!

Real-World Example: Cookie Manipulation

Consider an application that stores encrypted user roles in a cookie:

Plaintext: {"user":"alice","role":"user"}
Encrypted: [random-looking-base64-string]

An attacker who knows the plaintext structure can flip bits to change:

{"user":"alice","role":"user"}
                              ↓ (flip bits)
{"user":"alice","role":"admin"}

Result: Privilege escalation without knowing the encryption key!

Vulnerable Code Example

from Crypto.Cipher import AES
import base64

def create_session_cookie(username, role):
    plaintext = f"user={username};role={role}"
    # Pad to block size
    plaintext += " " * (16 - len(plaintext) % 16)
    
    cipher = AES.new(SECRET_KEY, AES.MODE_CBC, IV)
    ciphertext = cipher.encrypt(plaintext.encode())
    return base64.b64encode(IV + ciphertext).decode()

def verify_session_cookie(cookie):
    data = base64.b64decode(cookie)
    iv = data[:16]
    ciphertext = data[16:]
    
    cipher = AES.new(SECRET_KEY, AES.MODE_CBC, iv)
    plaintext = cipher.decrypt(ciphertext).decode()
    
    # INSECURE: No integrity check!
    # Attacker can flip bits in ciphertext
    return parse_cookie(plaintext)
Problem: The application trusts the decrypted data without verifying it hasn't been tampered with!

Secure Alternatives

1. Use Authenticated Encryption (AEAD)

from Crypto.Cipher import AES

# GCM mode provides both encryption and authentication
cipher = AES.new(key, AES.MODE_GCM)
ciphertext, tag = cipher.encrypt_and_digest(plaintext)

# On decryption, verify() will raise exception if tampered
plaintext = cipher.decrypt_and_verify(ciphertext, tag)

2. Encrypt-then-MAC

import hmac
import hashlib

# Encrypt first
ciphertext = encrypt_cbc(plaintext, key)

# Then create MAC
mac = hmac.new(mac_key, ciphertext, hashlib.sha256).digest()

# Verify MAC before decrypting
if not hmac.compare_digest(received_mac, computed_mac):
    raise ValueError("Integrity check failed!")
plaintext = decrypt_cbc(ciphertext, key)
Key Principles:
  • Never use encryption alone for data that must be trusted
  • Always use authenticated encryption (AES-GCM, ChaCha20-Poly1305)
  • If using CBC, add HMAC for integrity (Encrypt-then-MAC pattern)
  • Verify authentication tags BEFORE processing decrypted data

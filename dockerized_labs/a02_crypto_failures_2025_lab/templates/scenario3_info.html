Scenario 3: Weak Pseudorandom Number Generation

Vulnerability: Using non-cryptographic random for security tokens (CWE-330, CWE-338)

The Vulnerability

Many applications use random numbers for security-critical operations like generating session tokens, password reset codes, or authentication tokens. However, not all random number generators are created equal.

Weak Random Number Generators (like Python's random module) are predictable. They use algorithms that produce sequences of numbers that appear random but are actually deterministic based on an initial seed value.

If an attacker can:

  • Observe several generated values
  • Determine or guess the seed
  • Know what random number generator is being used

Then they can predict ALL future (and past) random values!

Real-World Example: 2008 Debian OpenSSL Bug

From 2006–2008, Debian's OpenSSL package had a bug that reduced the entropy for random number generation to only 32,767 possible values. This made SSH keys and SSL certificates generated during this period highly predictable.

  • Attackers could generate all possible keys in hours
  • Tens of thousands of servers were vulnerable
  • Required mass key regeneration across the internet

Vulnerable vs Secure Code

❌ Vulnerable – Time-Seeded PRNG

import random
import time

# INSECURE: Using time as seed
random.seed(time.time())

def generate_admin_token():
    # INSECURE: Predictable random number
    token = random.randint(100000, 999999)
    return f"ADMIN-{token}"

# Attacker can predict this!
admin_token = generate_admin_token()
Problems:
  • Using time.time() as seed means attacker knows seed value (within seconds)
  • random module is not cryptographically secure
  • Token space is only 900,000 possibilities (easily brute-forced)

✅ Secure – Cryptographically Secure Randomness

import secrets

def generate_admin_token():
    # SECURE: Using cryptographically secure random
    token = secrets.token_hex(16)  # 32 chars, 256 bits
    return f"ADMIN-{token}"

admin_token = generate_admin_token()

# Alternative: For random integers
secure_code = secrets.randbelow(1000000)
Improvements:
  • secrets module uses OS's cryptographically secure random source
  • No seed – uses system entropy pool
  • Much larger token space (2256 possibilities)
  • Unpredictable even if attacker observes many tokens

Best Practices

  • Never use random for security: Use secrets in Python, crypto.randomBytes() in Node.js, SecureRandom in Java
  • Avoid predictable seeds: Don't seed with timestamps, PIDs, or other guessable values
  • Use sufficient entropy: At least 128 bits for session IDs, 256 bits for keys
  • Don't roll your own: Use established cryptographic libraries
Challenge:

In the lab, you'll attack a system that uses time-based seeded random numbers to generate admin tokens. By predicting the seed, you can forge your own admin token!

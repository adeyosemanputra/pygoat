A2: Security Misconfiguration

What is this vulnerability?

Security misconfiguration happens when applications, servers, frameworks, cloud services, or containers are deployed with unsafe defaults, unnecessary features enabled, missing hardening, verbose errors in production, weak permissions, or exposed administrative interfaces. It is one of the most common issues because secure configuration must be repeated for every environment and release.

Real World Example

The 2017 Equifax breach involved an unpatched Apache Struts instance on a public-facing server. Attackers exploited a known vulnerability while the organization had not applied patches or aligned configuration with vendor security guidance—an operational and configuration failure rather than a novel coding bug.

Impact

  • Full compromise of applications or supporting infrastructure
  • Exposure of secrets, backups, debug output, or internal documentation
  • Easier lateral movement after an initial foothold

Mitigation

  • Use hardened baselines, infrastructure-as-code, and automated configuration checks
  • Disable debug modes, default accounts, and unused features in production
  • Apply least privilege, patch regularly, and review cloud and firewall rules

A6: Insecure Design

What is this vulnerability?

Insecure design means missing or ineffective security controls in requirements and architecture. Unlike a single coding mistake, the system is built without adequate threat modeling, separation of duties, or enforcement of business rules at the design level, so flaws persist across the codebase.

Real World Example

Account recovery flows that rely only on easily guessed knowledge-based questions, or APIs that trust the client to send the correct user role or price, illustrate insecure design: the code may work as specified but fails against realistic attackers.

Impact

  • Systemic fraud or abuse that spans multiple features
  • Expensive redesign when issues are discovered after release
  • Repeated incidents because root causes are architectural

Mitigation

  • Threat model features early and document abuse cases alongside use cases
  • Apply secure design patterns: least privilege, defense in depth, and server-side enforcement
  • Review designs with security stakeholders before implementation and test negative cases

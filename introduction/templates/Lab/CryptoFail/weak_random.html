Scenario 3: Insufficient Randomness

The Vulnerability

Using predictable or weak random number generators for security-critical operations is a common cryptographic failure. When applications use random.random() or similar functions instead of cryptographically secure random generators, attackers can predict tokens, session IDs, and password reset links.

Common insecure randomness scenarios:

  • Session tokens generated with random.random()
  • Password reset tokens using timestamp-based seeds
  • API keys generated with predictable patterns
  • CSRF tokens using weak randomness
  • Cryptographic keys derived from insufficient entropy

Attack Scenario - Session Token Prediction:
  1. Reconnaissance: Attacker creates multiple accounts and collects session tokens
  2. Pattern Analysis: Discovers tokens are generated using weak PRNG
  3. Seed Discovery: Determines the seed (often based on timestamp)
  4. Token Prediction: Predicts future or existing session tokens
  5. Session Hijacking: Uses predicted tokens to impersonate legitimate users
Real-World Example:
2008 Debian OpenSSL Bug: A bug in Debian's OpenSSL package reduced entropy from billions of possibilities to just 32,768 possible keys. This made SSH keys, SSL certificates, and other cryptographic material predictable and easily cracked. 

# VULNERABLE CODE - DO NOT USE
import random
import string
import time

def generate_session_token():
    # DANGER: Using random.random() which is predictable!
    random.seed(int(time.time()))  # Even worse - predictable seed
    token = ''.join(random.choices(string.ascii_letters + string.digits, k=32))
    return token

def generate_password_reset_token(user_id):
    # DANGER: Using user_id and timestamp - highly predictable!
    random.seed(user_id + int(time.time()))
    token = ''.join(random.choices(string.hexdigits, k=16))
    return token

# Example usage
session = generate_session_token()
# Attacker can predict this by guessing the timestamp!
Why This is Dangerous:
  • random.random() is NOT cryptographically secure
  • Same seed = same sequence of "random" numbers
  • Timestamps are easily guessable (within seconds)
  • Attackers can brute force small seed spaces

# SECURE CODE - Recommended Approach
import secrets
import os

def generate_session_token():
    # Uses cryptographically secure random generator
    token = secrets.token_urlsafe(32)  # 32 bytes = 256 bits
    return token

def generate_password_reset_token():
    # Alternative: using os.urandom
    token = secrets.token_hex(16)  # 16 bytes = 128 bits
    return token

def generate_api_key():
    # For API keys, use longer tokens
    api_key = secrets.token_urlsafe(48)  # 384 bits
    return api_key

# UUID4 is also acceptable (uses os.urandom internally)
import uuid
session_id = str(uuid.uuid4())

Your Mission

Challenge: The lab demonstrates a vulnerable token generation system. Your tasks:
  1. Generate multiple session tokens and observe the pattern
  2. Predict a future token based on the pattern
  3. Use the predicted token to gain unauthorized access
  4. Retrieve the flag

Mitigation

Best Practices for Random Number Generation:

  • Use secrets module for all security-critical randomness
  • Never use random.random(), random.randint() for security
  • Use os.urandom() or secrets for cryptographic keys
  • Token length: minimum 128 bits (16 bytes) for session tokens
  • API keys: minimum 256 bits (32 bytes)
  • Never use predictable seeds (timestamps, user IDs, counters)
  • Implement token expiration and rotation policies

Detection Methods

How to identify weak randomness:

  • Collect multiple tokens/IDs from the application
  • Analyze for patterns using statistical tests (Chi-square, runs test)
  • Check for sequential or time-based patterns
  • Use tools like ent to measure entropy
  • Review source code for use of random module

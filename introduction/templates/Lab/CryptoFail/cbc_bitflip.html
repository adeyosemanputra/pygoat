Scenario 4: CBC Bit-Flipping Attack (Advanced)

The Vulnerability

Using encryption modes like AES-CBC without integrity protection (HMAC or authenticated encryption) makes the ciphertext vulnerable to manipulation. Attackers can flip bits in the ciphertext to cause predictable changes in the plaintext after decryption, leading to privilege escalation and unauthorized access.

Common vulnerable patterns:

  • Encrypting session data/cookies with CBC mode only
  • No HMAC or message authentication code (MAC)
  • Using AES-CBC instead of AES-GCM (authenticated encryption)
  • Storing user roles/permissions in encrypted cookies without integrity checks

Cipher Block Chaining (CBC):

In CBC mode, each plaintext block is XORed with the previous ciphertext block before encryption:

C[0] = Encrypt(P[0] XOR IV)
C[1] = Encrypt(P[1] XOR C[0])
C[2] = Encrypt(P[2] XOR C[1])

Decryption process:

P[0] = Decrypt(C[0]) XOR IV
P[1] = Decrypt(C[1]) XOR C[0]
P[2] = Decrypt(C[2]) XOR C[1]
The Vulnerability:

If we modify C[0], it changes P[1] in a predictable way through the XOR operation. Without integrity protection, the attacker can manipulate ciphertext to change plaintext!

Bit-Flipping Attack Steps:
  1. Obtain encrypted cookie: Create account with known data (e.g., user=bob;admin=0)
  2. Identify target bits: Know which bytes to flip (admin=0 to admin=1)
  3. Calculate flip mask: XOR manipulation: 0x30 XOR 0x30 XOR 0x31 = 0x01
  4. Modify ciphertext: Flip the corresponding bit in the previous ciphertext block
  5. Submit modified cookie: Server decrypts and processes tampered data
  6. Privilege escalation: Now authenticated as admin!
Example Scenario:

Original plaintext: {"user":"bob","admin":false}

After bit-flipping: {"user":"bob","admin":true}

The attacker becomes admin without knowing the encryption key!


# VULNERABLE CODE - Encryption without authentication
from Crypto.Cipher import AES
from Crypto.Util.Padding import pad, unpad
import base64

KEY = b'sixteen_byte_key'
IV = b'sixteen_byte_iv!'

def create_session_cookie(username, is_admin):
    # Create session data
    data = f"user={username};admin={1 if is_admin else 0}"
    
    # Encrypt with CBC mode (NO HMAC!)
    cipher = AES.new(KEY, AES.MODE_CBC, IV)
    ciphertext = cipher.encrypt(pad(data.encode(), AES.block_size))
    
    # Return base64 encoded cookie
    return base64.b64encode(ciphertext).decode()

def validate_session_cookie(cookie):
    # Decrypt cookie
    ciphertext = base64.b64decode(cookie)
    cipher = AES.new(KEY, AES.MODE_CBC, IV)
    plaintext = unpad(cipher.decrypt(ciphertext), AES.block_size)
    
    # Parse session data (vulnerable - no integrity check!)
    data = plaintext.decode()
    is_admin = 'admin=1' in data
    
    return is_admin  # Attacker can manipulate this!
Why This Fails:
  • No HMAC or MAC to verify ciphertext integrity
  • Modifications to ciphertext go undetected
  • XOR properties allow predictable bit flips
  • First block (IV) can be manipulated to affect second block plaintext
Option 1: Use Authenticated Encryption (AES-GCM)

# SECURE - Using AES-GCM (authenticated encryption)
from Crypto.Cipher import AES
import base64
import os

def create_session_cookie(username, is_admin):
    data = f"user={username};admin={1 if is_admin else 0}"
    
    # Generate random nonce for each encryption
    nonce = os.urandom(12)
    
    # Use GCM mode (provides both encryption and authentication)
    cipher = AES.new(KEY, AES.MODE_GCM, nonce=nonce)
    ciphertext, tag = cipher.encrypt_and_digest(data.encode())
    
    # Include nonce and tag with ciphertext
    cookie_data = nonce + tag + ciphertext
    return base64.b64encode(cookie_data).decode()

def validate_session_cookie(cookie):
    cookie_data = base64.b64decode(cookie)
    
    nonce = cookie_data[:12]
    tag = cookie_data[12:28]
    ciphertext = cookie_data[28:]
    
    cipher = AES.new(KEY, AES.MODE_GCM, nonce=nonce)
    
    try:
        # Verify AND decrypt (will fail if tampered!)
        plaintext = cipher.decrypt_and_verify(ciphertext, tag)
        data = plaintext.decode()
        return 'admin=1' in data
    except ValueError:
        # Tampering detected!
        return False
Option 2: Encrypt-then-MAC

# SECURE - CBC with HMAC (Encrypt-then-MAC)
import hmac
import hashlib

def create_session_cookie(username, is_admin):
    data = f"user={username};admin={1 if is_admin else 0}"
    
    # Encrypt with CBC
    cipher = AES.new(KEY, AES.MODE_CBC, IV)
    ciphertext = cipher.encrypt(pad(data.encode(), AES.block_size))
    
    # Create HMAC of ciphertext
    mac = hmac.new(MAC_KEY, ciphertext, hashlib.sha256).digest()
    
    # Return ciphertext + MAC
    cookie_data = ciphertext + mac
    return base64.b64encode(cookie_data).decode()

def validate_session_cookie(cookie):
    cookie_data = base64.b64decode(cookie)
    
    ciphertext = cookie_data[:-32]  # All but last 32 bytes
    received_mac = cookie_data[-32:]  # Last 32 bytes
    
    # Verify HMAC first!
    expected_mac = hmac.new(MAC_KEY, ciphertext, hashlib.sha256).digest()
    
    if not hmac.compare_digest(received_mac, expected_mac):
        raise ValueError("MAC verification failed - tampering detected!")
    
    # Only decrypt if MAC is valid
    cipher = AES.new(KEY, AES.MODE_CBC, IV)
    plaintext = unpad(cipher.decrypt(ciphertext), AES.block_size)
    
    data = plaintext.decode()
    return 'admin=1' in data

Your Mission

Challenge: This lab has a vulnerable session cookie system using CBC without authentication. Your tasks:
  1. Create a regular user account
  2. Obtain your encrypted session cookie
  3. Perform a bit-flipping attack to change admin=0 to admin=1
  4. Submit the modified cookie
  5. Access the admin panel and retrieve the flag

Mitigation

Best Practices:

  • Use AES-GCM or ChaCha20-Poly1305 for authenticated encryption
  • If using CBC, always implement Encrypt-then-MAC (not MAC-then-Encrypt)
  • Use separate keys for encryption and MAC
  • Validate MAC before decryption
  • Use constant-time comparison for MAC verification
  • Never trust client-side data without cryptographic verification
  • Consider using signed JWTs instead of encrypted cookies

A02:2025 Cryptographic Failures

What are Cryptographic Failures?

OWASP Top 10:2025 ranks Cryptographic Failures as #2, evolving from "Sensitive Data Exposure" to focus specifically on cryptographic implementation failures that lead to data breaches.

Cryptographic failures occur when applications improperly protect sensitive data through:

  • Weak encryption algorithms - Using deprecated algorithms like DES, 3DES, or MD5
  • Insecure key management - Hardcoded keys, weak key derivation, or plaintext storage
  • Insufficient randomness - Using predictable random number generators for security-critical operations
  • Lack of integrity protection - Encryption without authentication (e.g., CBC without HMAC)
  • Improper algorithm implementation - Custom crypto, weak parameters, or misuse of libraries

Real-World Impact

Major breaches involving cryptographic failures:

  • LinkedIn (2012) - 117M passwords stolen, hashed with unsalted SHA-1
  • Adobe (2013) - 150M records encrypted with faulty ECB mode encryption
  • Equifax (2017) - Weak cryptographic practices contributed to 147M records exposed
  • Marriott (2018) - Poor encryption key management led to 500M guest records compromised

CWE Mappings

A02:2025 maps to several Common Weakness Enumerations:

  • CWE-259 - Use of Hard-coded Password
  • CWE-327 - Use of a Broken or Risky Cryptographic Algorithm
  • CWE-330 - Use of Insufficiently Random Values
  • CWE-338 - Use of Cryptographically Weak Pseudo-Random Number Generator (PRNG)

Lab Scenarios

📌 Scenario 1: Weak Encryption Algorithm

Vulnerability: Application uses MD5 for password hashing instead of modern algorithms.
Impact: Passwords can be cracked using rainbow tables or brute force.

📌 Scenario 2: Insecure Cryptographic Storage

Vulnerability: Encryption keys hardcoded in source code or stored in plaintext.
Impact: Complete compromise of all "encrypted" data.

📌 Scenario 3: Insufficient Randomness (Coming Soon)

Vulnerability: Using predictable random number generators for tokens/sessions.
Impact: Session hijacking, account takeover.

Mitigation Best Practices

Follow these guidelines to prevent cryptographic failures:

  • ✅ Use bcrypt, Argon2, or PBKDF2 for password hashing (never MD5/SHA-1)
  • ✅ Use AES-256-GCM for encryption with authenticated encryption
  • ✅ Use secrets module (Python) for generating cryptographically secure random values
  • ✅ Never hardcode encryption keys - use environment variables or key vaults
  • ✅ Always transmit sensitive data over TLS 1.2+
  • ✅ Implement proper key rotation policies
  • ✅ Use established crypto libraries - never roll your own crypto!

Regulatory Requirements

Proper cryptography is mandated by:

  • GDPR (Article 32) - Encryption of personal data
  • PCI-DSS (Requirement 3) - Protect stored cardholder data
  • HIPAA - Encryption of ePHI (electronic Protected Health Information)
  • NIST - Cryptographic standards and guidelines

Scenario 2: Insecure Cryptographic Storage

The Vulnerability

This scenario demonstrates one of the most critical crypto failures: hardcoded encryption keys. Even if applications use strong encryption algorithms like AES-256, hardcoding keys completely defeats the purpose.

Common insecure storage practices include:

  • Hardcoded keys directly in source code
  • Encryption keys stored in configuration files committed to Git
  • Keys in environment variables without proper access control
  • Using the same key for encryption and authentication
  • No key rotation policy

Attack Scenario:
  1. Recon: Attacker discovers the codebase (GitHub, GitLab, insider access)
  2. Key Discovery: Finds hardcoded encryption key in source code: 
    SECRET_KEY = "my_secret_key_12345"  # Found in settings.py
  3. Data Access: Gains read access to encrypted database (SQL injection, backup leak)
  4. Decryption: Uses the hardcoded key to decrypt ALL encrypted data
  5. Impact: Complete compromise of confidentiality - credit cards, SSNs, medical records, etc.
Real Example: In 2019, a major company had AWS credentials hardcoded in their public GitHub repository. Attackers discovered this, gained access to their S3 buckets, and exfiltrated 100M+ customer records. 

# ❌ VULNERABLE CODE - DO NOT USE IN PRODUCTION
from cryptography.fernet import Fernet

# DANGER: Hardcoded encryption key!
ENCRYPTION_KEY = b'your-secret-key-here-1234567890ab='

def encrypt_sensitive_data(data):
    cipher = Fernet(ENCRYPTION_KEY)  # Using hardcoded key
    encrypted = cipher.encrypt(data.encode())
    return encrypted

def decrypt_sensitive_data(encrypted_data):
    cipher = Fernet(ENCRYPTION_KEY)  # Same hardcoded key
    decrypted = cipher.decrypt(encrypted_data)
    return decrypted.decode()

# Example: Encrypting credit card
credit_card = "4532-1234-5678-9010"
encrypted_cc = encrypt_sensitive_data(credit_card)
# This key is in the source code - anyone with code access can decrypt!
Why This is Dangerous:
  • ✗ Source code often ends up in version control (Git history)
  • ✗ Developers may accidentally commit to public repositories
  • ✗ Code is shared with contractors, offshore teams, etc.
  • ✗ Cannot rotate keys without code changes and redeployment

# ✅ SECURE CODE - Recommended Approach
import os
from cryptography.fernet import Fernet

# Method 1: Environment Variables (Better)
ENCRYPTION_KEY = os.environ.get('ENCRYPTION_KEY')
if not ENCRYPTION_KEY:
    raise ValueError("ENCRYPTION_KEY environment variable not set!")

# Method 2: Secrets Manager (Best)
import boto3

def get_encryption_key():
    client = boto3.client('secretsmanager')
    response = client.get_secret_value(SecretId='prod/encryption/key')
    return response['SecretString'].encode()

ENCRYPTION_KEY = get_encryption_key()

# Method 3: Key Derivation from Master Password (Alternative)
from cryptography.hazmat.primitives.kdf.pbkdf2 import PBKDF2HMAC
from cryptography.hazmat.primitives import hashes
import base64

def derive_key_from_password(password, salt):
    kdf = PBKDF2HMAC(
        algorithm=hashes.SHA256(),
        length=32,
        salt=salt,
        iterations=100000,
    )
    key = base64.urlsafe_b64encode(kdf.derive(password.encode()))
    return key

# Use derived key instead of hardcoded
ENCRYPTION_KEY = derive_key_from_password(
    password=os.environ['MASTER_PASSWORD'],
    salt=os.environ['ENCRYPTION_SALT'].encode()
)

Prevent accidental commits of secrets using these tools:

  • git-secrets - AWS tool to prevent committing secrets 
    git secrets --install
git secrets --register-aws
  • TruffleHog - Searches Git history for high entropy strings 
    trufflehog git https://github.com/your-repo --only-verified
  • GitHub Secret Scanning - Automatic scanning for known token patterns
  • pre-commit hooks - Block commits with potential secrets

Mitigation Strategies

Best Practices for Key Management:
Approach Security Level Use Case
Hardcoded in code ❌ Insecure Never use
Config file in repo ❌ Insecure Never use
Environment variables ⚠️ Basic Development only
Key derivation (PBKDF2) ✅ Good Small scale production
Secrets Manager (AWS/Azure) ✅ Excellent Production (recommended)
Hardware Security Module (HSM) ✅ Best High security requirements

Additional Recommendations

  • ✅ Implement key rotation policies (e.g., rotate every 90 days)
  • ✅ Use separate keys for encryption and authentication
  • ✅ Store keys in dedicated secrets management systems
  • ✅ Apply principle of least privilege for key access
  • ✅ Use .gitignore to exclude config files with secrets
  • ✅ Scan code with secret detection tools in CI/CD pipeline
  • ✅ Never log encryption keys or sensitive data

  • NIST Key Management Guidelines
